# Initial Design Brief

## Project Name

**Field Notes**

GitHub repo name: `scrolling-portfolio-site`

## Concept

**Field Notes From a Systems Traveller**

A cinematic portfolio for a travelling software engineer and site reliability engineer. The site should feel like entering a living landscape where nature, infrastructure, code, and reliability are part of the same world.

This is inspired by high-end scrolling sites, especially the feeling of Borealis: immersive, system-like, atmospheric, and deliberate. The direction should not become a clone. It should become a personal world for Tushar.

## Core Feeling

Calm, technical, cinematic, alive.

The visitor should feel:

- This person thinks in systems.
- This person can handle complexity.
- This person cares about craft.
- This person has a distinct creative point of view.

## World Metaphor

Natural systems become software systems:

- Mountains are infrastructure.
- Rivers are data flows.
- Weather is load, failure, and incident pressure.
- Stars are logs, metrics, traces, and signals.
- Trails are deployment paths.
- Basecamp is the personal intro.
- Expeditions are project case studies.

## Opening Sequence

The first experience can feel like a route/system booting up:

```text
INITIALIZING ROUTE
Mapping services...
Checking signal...
Warming terminal...
Syncing coordinates...
Entering the field...
```

After the boot, the visitor lands in the opening scene:

```text
Tushar
Software Engineer / Site Reliability Engineer

Building reliable systems from shifting terrain.
```

## First Scroll Journey

The first version should be a short, polished scroll experience:

1. Boot sequence.
2. Basecamp intro.
3. Terrain/skills preview.
4. Expeditions/projects preview.
5. Next coordinates/contact.

The goal of phase 1 is not to include everything. The goal is to make the first 30 seconds memorable.

## Sections

### Basecamp

The identity section. Short and confident.

Possible line:

```text
Traveller. Software Engineer. Site Reliability Engineer.
I build systems that stay steady when the terrain changes.
```

### The Terrain

Skills represented as parts of a landscape:

- Backend as bedrock.
- Infrastructure as mountain routes.
- Observability as stars and signals.
- Reliability as weather patterns.
- Automation as rivers and flows.

### Expeditions

Projects as places on a route. Each project should eventually have:

- Problem.
- Constraints.
- Route taken.
- Technical choices.
- Outcome.
- Link to code or demo.

### Incidents & Recovery

The SRE-specific section. This can become one of the strongest differentiators.

Possible line:

```text
Every system has weather.
I build for the storm.
```

### Toolkit

Tools presented as expedition gear:

- Linux
- Docker
- Kubernetes
- Cloud
- CI/CD
- Monitoring
- Databases
- Python
- JavaScript
- Go

### Next Coordinates

Contact and links:

- GitHub
- LinkedIn
- Resume
- Email

## Design Principles

- Build a world, not a template.
- Keep the copy short and confident.
- Use motion to explain structure, not to decorate every pixel.
- Nature should feel connected to systems, not like random wallpaper.
- The first version should be small enough to finish.
- Performance matters because a slow portfolio weakens the reliability story.

## Phase Plan

### Phase 1: Basecamp Prototype

- Boot/loading sequence.
- Full-screen opening scene.
- Name and role.
- Smooth scroll into 2-3 sections.
- Early visual language for nature plus systems.

### Phase 2: Terrain System

- Interactive skill map.
- Scroll-triggered reveals.
- Lightweight motion and micro-interactions.

### Phase 3: Project Expeditions

- First real project case studies.
- Each project gets a route and story.
- Include screenshots, architecture notes, code links, and impact.

### Phase 4: SRE Storytelling

- Incident timeline.
- Observability visuals.
- Recovery and postmortem thinking.
- Scaling and operational decisions.

### Phase 5: Polish and Deploy

- Mobile layout.
- Performance pass.
- Accessibility pass.
- SEO and metadata.
- Deployment.
